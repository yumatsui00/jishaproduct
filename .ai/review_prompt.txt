## AGENTS.md
# AGENTS.md (Repository Guidelines)
AI Coding and Review Agent Guidelines

---

## 1. Purpose
Act as the coding/review agent for this repository. Implement only the task requirements while following Repository Guidelines and AI/task/. Prioritize precision, consistency, and safety.

---

## 2. Global Principles
### Architecture & Project Rules
- Structure: `src/` (App Router under `src/app`, shared UI in `src/components`, utilities `src/utils`, types `src/types`), `logic/` (domain rules), `db/` (sample data in JSON format), `assets/` (static; translations in `assets/translations/<lang>.ts`), `tests/`, `scripts/`, `AI/`, `AI/task/` (work items), `requirements.txt`. Use absolute imports `@/*`; do not write outside these directories.
- Pages contain no business logic; domain UI lives in `src/components/<domain>`; logic lives in `logic/`; API routes are placed only under `src/app/api/**/route.ts` and call `logic/`. Frontend uses shadcn/ui as the primary UI kit; wrap components in `src/components/ui`.
- UI text is never hardcoded; use keys `<domain>.<feature>.<element>`. Prefer Tailwind composition over custom CSS.
- Shared cross-layer types must live under `src/types/`. The canonical `Result<T>` type is defined in `src/types/result.ts` and imported via `@/types/result`.

### Coding Philosophy
- TypeScript strict must stay enabled. Enforce DRY/KISS/YAGNI, thin abstractions, and single responsibility per file. CRUD-style functions in `logic/` or `src/utils/` should aim for ≤40 lines and ≤80 chars per line (UI exempt). All functions require JSDoc (summary, params, returns, errors if non-Result). Comments in English, focus on “why.”
- Naming: Components/types PascalCase; logic/utils camelCase; DB helpers snake_case; API routes `route.ts`; CSS kebab-case.

### Reasoning & Depth

# 5. Logic Specification

## 5.1 Inputs

The feature receives the following inputs:

- Page render request for `/`
- Filter UI state:
  - `selectedChallenges: string[]`
  - `selectedIndustries: string[]`
  - `selectedPhases: string[]`
- Pagination state:
  - `page: number`
- Existing JSON records loaded from `db/case-studies.json`

## 5.2 Processing Flow (Strict Order)

1. Render the landing page with the hero section and the case-study section shell.
2. Initialize the case-study section with page `1`, no selected filters, and page size `20`.
3. When the section loads, request case-study data from `/api/case-studies` using the current applied filters and page.
4. In the API route, parse query parameters into `CaseStudySearchParams`.
5. Validate `page`:
   - if missing, use `1`
   - if invalid, non-numeric, or less than `1`, use `1`
6. Normalize filter arrays:
   - missing values become empty arrays
   - duplicate selected values are removed
   - unknown values are ignored
7. Call `logic/caseStudy/getCaseStudies.ts`.
8. In logic, request all records from the repository.
9. Build `availableFilters` from the full dataset.
10. Filter records:
    - `challenge`: include a record when `selectedChallenges` is empty or its `challenge` matches one selected value
    - `industry`: include a record when `selectedIndustries` is empty or its `industry` matches one selected value
    - `phase`: include a record when `selectedPhases` is empty or its `phase` matches one selected value
11. Count filtered results.
12. Compute `totalPages` as at least `1`.
13. If requested `page` exceeds `totalPages`, return the last available page of results.
14. Slice the filtered list for the current page with a fixed page size of `20`.
15. Return a successful `Result<CaseStudyListResponse>` from logic.
16. In the API route, convert the `Result` into a JSON HTTP response.
17. In the UI, update the displayed cards and pagination metadata from the API response.
18. When users change checkboxes, update local draft filter state only.
19. Only when users press the search button:
    - copy draft filters into applied filters
    - reset page to `1`
    - fetch results again
20. When users press previous or next:
    - move one page backward or forward within valid bounds
    - fetch results using the current applied filters

## 5.3 Outputs

The feature must produce:

- A landing page with the required two content sections
- A `Result<CaseStudyListResponse>` from `logic/caseStudy/getCaseStudies.ts`
- A JSON API response containing paginated case-study data
- Updated filter and pagination UI state
- A rendered list of up to 20 case-study cards

## 5.4 Error Handling

All errors must follow the project’s unified `Result<T>` pattern.

Expected error types:

- `IO_ERROR`: JSON file missing, unreadable, or invalid
- `VALIDATION_ERROR`: only for unrecoverable malformed query input after normalization

UI handling rules:

- If the initial or subsequent fetch fails, keep the page rendered and show a simple translated error message inside the case-study section
- Do not clear the current checkbox draft state on fetch failure
- Disable pagination buttons when navigation is not possible
- Disable the search button only while the current request is in flight
- Logic and repository must not throw; use `console.error` and return an error `Result`
